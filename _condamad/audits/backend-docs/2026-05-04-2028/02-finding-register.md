## F-001 - Non-canonical LLM prose should move out of `backend/docs`

- Severity: Medium
- Confidence: High
- Category: boundary-violation
- Domain: backend-docs
- Evidence: E-003, E-004, E-005, E-009.
- Expected rule: `backend/docs/` should keep backend-owned generated docs, executable registries, and local governance artifacts; human non-canonical prose belongs under root `docs/`.
- Actual state: `llm-runtime-source-of-truth.md`, `llm-canonical-consumption-rebuild.md`, and `llm-db-governance.md` are classified as `historical-note` or `human-runbook` with non-canonical status, but still live under `backend/docs/`.
- Impact: `backend/docs/` still owns non-canonical LLM prose that fits the root `docs/` human documentation role better than the backend executable/generated docs role.
- Recommended action: Move retained non-canonical LLM prose to root `docs/llm/`, then update `ownership-index.md`, LLM governance rows, registry references, and guards.
- Story candidate: yes
- Suggested archetype: documentation-boundary-convergence / legacy-surface-removal

## F-002 - Entitlement historical note remains in backend docs

- Severity: Medium
- Confidence: High
- Category: legacy-surface
- Domain: backend-docs
- Evidence: E-004, E-007, E-009.
- Expected rule: historical architecture notes should not remain beside active backend generated/executable artifacts unless they are still backend-owned source contracts.
- Actual state: `entitlements-canonical-platform.md` is explicitly `historical-note`, and tests guard that it is not the active source of truth, but the title and filename still present it as canonical platform architecture.
- Impact: A historical entitlement document with canonical-looking title/name can be mistaken for active backend source of truth while living beside generated/executable backend docs.
- Recommended action: Move it to root `docs/architecture/` if retained, or delete it after user decision and extraction of still-useful decommission/security warnings.
- Story candidate: yes
- Suggested archetype: historical-doc-reposition-or-delete

## F-003 - Generated/executable LLM assets must be preserved

- Severity: Info
- Confidence: High
- Category: observability-gap
- Domain: backend-docs
- Evidence: E-004, E-006.
- Expected rule: reorganization must distinguish passive prose from artifacts consumed or generated by code/tests.
- Actual state: `llm-model-structure.md` and `llm-db-cleanup-registry.json` are classified and guarded as generated/executable assets.
- Impact: Reorganization pressure could accidentally move/delete guarded LLM assets that are not passive legacy prose.
- Recommended action: Keep `llm-model-structure.md` and `llm-db-cleanup-registry.json` in `backend/docs/` unless their code consumers and tests are migrated deliberately.
- Story candidate: no
- Suggested archetype: monitor

## F-004 - Calibration legacy surface is resolved

- Severity: Info
- Confidence: High
- Category: legacy-surface
- Domain: backend-docs
- Evidence: E-001, E-002, E-008.
- Expected rule: generated calibration artifacts have one canonical location.
- Actual state: `backend/docs/calibration` is absent and root `docs/calibration` is guarded as canonical.
- Impact: The previous stale calibration artifact is resolved; regression risk is reintroduction of `backend/docs/calibration`.
- Recommended action: Preserve the calibration location guard and root `docs/calibration` canonical path.
- Story candidate: no
- Suggested archetype: monitor
