## Objective

Implement the canonical backend-domain builder that produces `structured_facts_v1` from the existing canonical runtime and closest projection owners.

The implementation must reuse or extend the closest existing backend owner before creating a new module, produce deterministic facts ready for canonical
hashing, keep interpretive signals separate, exclude narrative text and keep the projection out of public API surfaces.

## Target State

- `structured_facts_v1` is generated by one tested builder or a clearly extended existing builder.
- The builder emits a deterministic, JSON-serializable payload ready for canonical hashing by persistence or audit layers.
- The payload separates structural facts from interpretive signals and excludes narration, prompt text, advice and LLM output.
- Missing optional runtime data is represented by stable empty collections, explicit nulls or omitted optional fields according to the CS-256 contract.
- The builder reuses `ChartInterpretationInputBuilder`, `AINarrativeInputBuilder`, runtime contracts or the closest existing projection owner.
- Internal primitives such as raw `ChartObjectRuntimeData`, raw `chart_objects`, debug traces and runtime traces are not exposed publicly.
- No public API route, frontend source, database migration, LLM narration or `beginner_summary_v1` implementation is introduced.

## Contract Shape

- Contract type:
  - Backend-domain runtime builder payload for a stable factual projection.
- Fields:
  - `projection_id`: exact value `structured_facts_v1`.
  - `contract_version`: exact stable version from CS-256.
  - `source_versions`: runtime, interpretation input, graph trace or reference versions used by the builder.
  - `structural_facts`: deterministic fact collections for positions, houses, major aspects and source metadata.
  - `interpretive_signals`: deterministic signal identifiers only, separated from structural facts.
  - `dominants`: deterministic dominance facts from existing runtime outputs.
  - `hash_input`: canonical serialization boundary or directly hashable payload section.
  - `missing_data`: deterministic representation for unavailable optional runtime data.
  - `excluded_surfaces`: raw runtime objects, narration, prompts, advice and LLM output.
- Required fields:
  - `projection_id`
  - `contract_version`
  - `source_versions`
  - `structural_facts`
  - `interpretive_signals`
  - `hash_input`
  - `missing_data`
- Optional fields:
  - `dominants` only when canonical dominance runtime data is available.
- Status codes:
  - none; no HTTP endpoint implementation is authorized.
- Serialization names:
  - emitted JSON keys must be stable snake_case names from the builder contract.
- Frontend type impact:
  - none; no frontend source or generated client is touched.
- Generated contract impact:
  - `app.openapi()` must not expose `structured_facts_v1` from this story.
