## Objective

Implement the canonical backend-domain builder that produces `client_interpretation_projection_v1` for free, basic and premium plans.

The implementation must reuse or extend the closest existing interpretation owner, build plan-specific sections from authorized facts and signals,
apply the entitlement policy, attach disclaimer references, prepare basic and premium audit-ready content, and avoid any LLM provider or final prose work.

## Target State

- `client_interpretation_projection_v1` is generated by one tested backend builder or a clearly extended existing projection owner.
- The builder consumes `structured_facts_v1` and authorized interpretive signals instead of recalculating raw astrology runtime data.
- Free, basic and premium outputs differ by depth, sections, personalization and audit readiness, not by exposing raw runtime payloads.
- Plan entitlement denial returns a controlled `plan_insufficient` result or error contract without technical payload leakage.
- Disclaimer references from CS-284 are attached as stable application-controlled codes or references.
- Basic and premium outputs contain audit-ready structured inputs for `narrative_answer_audit_v1` without exposing audit internals.
- No public API route, frontend source, database migration, prompt template, provider integration or expert projection is introduced.

## Contract Shape

- Contract type:
  - Backend-domain runtime builder payload for a deterministic B2C interpretation projection by plan.
- Fields:
  - `projection_id`: exact value `client_interpretation_projection_v1`.
  - `source_projection_id`: exact value `structured_facts_v1`.
  - `plan`: one of `free`, `basic` or `premium`.
  - `state`: controlled public state such as `normal`, `degraded`, `unavailable` or `plan_insufficient`.
  - `sections`: plan-specific client interpretation sections derived from facts and signals.
  - `support_elements`: vulgarized highlights, confidence wording, source labels or display hints allowed by plan.
  - `interpretive_signals`: authorized pre-narrative signal identifiers, not raw runtime objects.
  - `disclaimer_codes`: static application-controlled disclaimer references.
  - `audit_input`: audit-ready structured summary for basic and premium only.
  - `missing_data`: controlled public reasons for degraded or unavailable states.
  - `excluded_surfaces`: raw runtime objects, prompts, provider responses, final prose, debug traces and audit internals.
- Required fields:
  - `projection_id`
  - `source_projection_id`
  - `plan`
  - `state`
  - `sections`
  - `support_elements`
  - `disclaimer_codes`
  - `excluded_surfaces`
- Optional fields:
  - `interpretive_signals` only when authorized by the plan contract.
  - `audit_input` only for basic and premium projections.
  - `missing_data` only for degraded or unavailable states.
- Status codes:
  - none; no HTTP endpoint implementation is authorized.
- Serialization names:
  - emitted JSON keys must be stable snake_case names from the projection contract.
- Frontend type impact:
  - none; no frontend source or generated client is touched.
- Generated contract impact:
  - `app.openapi()` must not expose `client_interpretation_projection_v1` from this story.
