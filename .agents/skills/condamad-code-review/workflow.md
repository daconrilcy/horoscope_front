# CONDAMAD Code Review Workflow

## Goal

Review implemented story changes adversarially and produce reviewer-grade
findings grounded in:

- the real repository diff;
- the story or CONDAMAD capsule;
- acceptance criteria and final evidence;
- validation commands actually run;
- DRY / No Legacy constraints;
- `_condamad/stories/regression-guardrails.md` shared invariants;
- security, data, error-handling, and regression risks.

This workflow is inspired by BMAD code review, but it is Codex-native:

- no mandatory human checkpoint when Codex can infer the safe next action;
- no mandatory subagents unless the current user explicitly authorizes agent
  delegation;
- persistent review evidence when a story capsule exists;
- findings-first output, compatible with Codex review conventions;
- explicit support for modern Codex-class long-context repository inspection and
  stronger self-critique loops.

## Source-of-Truth Precedence

Apply this order when instructions conflict:

1. System/developer safety rules and current tool constraints.
2. Applicable repository `AGENTS.md`.
3. Story acceptance criteria and explicit non-goals.
4. CONDAMAD capsule evidence and contracts.
5. Architecture/project docs and CI configuration.
6. Existing code and tests.
7. Current user instruction for review target, scope emphasis, and output style.
8. This skill's review doctrine.

Do not change acceptance criteria, story scope, or repository doctrine during a
review. The current user may narrow the review target or request a specific
focus, but must not lower severity, suppress findings, mark skipped checks as
passing, or override repository architecture and story acceptance criteria.

## Review Artifact

When a CONDAMAD capsule exists, write review output to:

```text
_condamad/stories/<story-key>/generated/11-code-review.md
```

`generated/11-code-review.md` is the latest review result. Replace the full file
on each review run. Do not append contradictory review runs to the same file. If
history is needed later, create timestamped files under `generated/reviews/`.

When no capsule exists but a story file exists, append findings only to an
allowed review section if the story contract permits it. Otherwise, keep the
review in chat and say that it was not persisted.

Creating or updating `_condamad/stories/regression-guardrails.md` is allowed as
a governance artifact when the registry is missing or a completed story
establishes a new durable invariant. Do not edit application code during review
unless the user explicitly asks for fixes.

Recommended review artifact structure:

```md
# CONDAMAD Code Review

## Review target
## Inputs reviewed
## Diff summary
## Review layers
## Findings
## Acceptance audit
## Validation audit
## DRY / No Legacy audit
## Commands run by reviewer
## Residual risks
## Verdict
```

## Workflow Steps

Follow these steps in order:

1. `steps/step-01-target-and-context.md`
2. `steps/step-02-adversarial-review.md`
3. `steps/step-03-triage-and-evidence.md`
4. `steps/step-04-report-and-next-action.md`

At each step, include applicable regression guardrail IDs in the acceptance and
validation audit. A missing guard, missing baseline, or unreviewed allowed
difference for an applicable invariant is a finding.

## Non-Negotiable Rules

- Review mode is read-first. Do not edit application code unless the user
  explicitly asks you to fix findings.
- Preserve the user's worktree. Do not revert unrelated changes.
- Use `rg` and targeted file reads before drawing conclusions.
- Findings must include file/line evidence whenever possible.
- Findings must lead the final answer, ordered by severity.
- Do not report style-only noise unless it creates maintainability, correctness,
  security, or test risk.
- Do not treat a skipped command as passing.
- Do not accept final evidence that lacks AC-to-validation mapping.
- For No Legacy stories, classify legacy/search hits; unclassified active hits
  are review findings.
- If subagents are not explicitly authorized by the current user, run the
  adversarial layers yourself sequentially.
- If subagents are explicitly authorized, use them only for read-only review
  layers and keep final triage ownership in the main Codex session.

## Reviewer Command Policy

The reviewer may run non-mutating commands, including:

- `git status --short`
- `git diff ...`
- `git diff --check`
- `git ls-files --others --exclude-standard`
- `rg ...`
- targeted test commands
- type checks
- lint checks in check-only mode

The reviewer must not run mutating commands unless the user explicitly asks for
fixes, including:

- formatters without `--check`;
- code generators;
- migrations;
- dependency updates;
- cleanup scripts;
- destructive git commands.

For Python repositories, prefer the repository command policy from `AGENTS.md`.
If a local venv exists, activate it when possible:

```powershell
# Windows PowerShell
.\.venv\Scripts\Activate.ps1
```

```bash
# POSIX shell
source .venv/bin/activate
```

When activation is uncertain, prefer invoking the venv interpreter directly:

```powershell
# Windows PowerShell
.\.venv\Scripts\python.exe -m pytest -q
```

```bash
# POSIX shell
.venv/bin/python -m pytest -q
```

In review mode, prefer check-only commands such as:

```powershell
ruff format --check .
ruff check .
pytest -q
```

Do not claim tests passed unless the exact command and environment were
recorded. Do not run `ruff format .` during review unless the user explicitly
asked to fix formatting.

## Final Verdict Values

Use exactly one:

- `BLOCKING`: Critical issue, High AC failure, security/data risk, active
  forbidden legacy, false validation claim, or missing validation evidence
  required by the story or definition of done.
- `CHANGES_REQUESTED`: material non-blocking defects require patching before
  acceptance.
- `ACCEPTABLE_WITH_LIMITATIONS`: no required patch found, but optional or
  full-regression validation could not be run and the residual risk is
  documented.
- `CLEAN`: no actionable findings and validation evidence is adequate.
