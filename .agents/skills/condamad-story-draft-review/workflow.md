# CONDAMAD Story Draft Review Workflow

## Goal

Review drafted CONDAMAD story contracts adversarially and produce reviewer-grade
findings grounded in:

- the story or CONDAMAD capsule;
- acceptance criteria and expected validation evidence;
- source audit finding/candidate when the story was generated from an audit;
- story validation and strict lint output;
- DRY / No Legacy story constraints;
- `_condamad/stories/regression-guardrails.md` shared invariants;
- scope, non-goals, review artifact path, and regression risks.

This workflow is inspired by BMAD-style review discipline, but it is
Codex-native:

- no mandatory human checkpoint when Codex can infer the safe next action;
- no mandatory subagents unless the current user explicitly authorizes agent
  delegation;
- persistent editorial review evidence when a story capsule exists;
- findings-first output, compatible with Codex review conventions;
- explicit support for modern Codex-class long-context repository inspection and
  stronger self-critique loops.

## Reference Loading Policy

For story-contract review, read these files from this skill only when needed:

1. `references/review-doctrine.md`
2. `references/finding-taxonomy.md`
3. `references/codex-modern-review-guidance.md`
4. `../condamad-regression-guardrails/SKILL.md`

For pre-implementation story-contract review, defer these broad references
unless the story text or validation output raises a concrete issue that needs
the specific doctrine. Use the story, tracker row, scoped guardrail IDs, and
validation/lint output first.

When reviewing a CONDAMAD capsule generated by `condamad-story-writer` or later
expanded by `condamad-dev-story`, use the story files as primary evidence:

```text
_condamad/stories/<story-key>/
  00-story.md
  generated/11-code-review.md
```

When the story references an audit finding or story candidate, also review that
source material and the latest same-domain audit or sibling stories when they
are available. The review must verify whether the story contract covers the
source, defines an explicitly bounded phase, blocks on a decision, or defers a
non-domain concern.

Before reviewing, ensure `_condamad/stories/regression-guardrails.md` exists.
If it is missing, create it from the `condamad-regression-guardrails` template,
then classify applicable invariants for the review target.

For normal review work, treat the guardrail registry as a global catalog, not a
review input to paste or scan wholesale. Prefer IDs already listed in the story
or capsule evidence, or use a scoped resolver/search for the exact paths,
routes, operations, and contracts under review.

## Source-of-Truth Precedence

Apply this order when instructions conflict:

1. System/developer safety rules and current tool constraints.
2. Applicable repository `AGENTS.md`.
3. Story acceptance criteria and explicit non-goals.
4. CONDAMAD capsule evidence and contracts.
5. Architecture/project docs only when the story cites them.
6. Existing code and tests only as context for story feasibility, not as a code
   review target.
7. Current user instruction for review target, scope emphasis, and output style.
8. This skill's review doctrine.

Do not change acceptance criteria, story scope, or repository doctrine during a
review. The current user may narrow the review target or request a specific
focus, but must not lower severity, suppress findings, mark skipped checks as
passing, or override repository architecture and story acceptance criteria.

## Editorial Review Artifact

When a CONDAMAD capsule exists, write review output to:

```text
_condamad/stories/<story-key>/generated/11-code-review.md
```

`generated/11-code-review.md` remains the latest review result for historical
compatibility. Its content is an editorial story review, not an application-code
review. Replace the full file on each review run. Do not append contradictory
review runs to the same file. If history is needed later, create timestamped
files under `generated/reviews/`.

When no capsule exists but a story file exists, append findings only to an
allowed review section if the story contract permits it. Otherwise, keep the
review in chat and say that it was not persisted.

Creating `_condamad/stories/regression-guardrails.md` is allowed only when the
registry is missing. Updating existing registry rows is registry-maintenance
work and is allowed only when the user explicitly asks for it or when the
current story itself created a new durable invariant that must be recorded. Do
not edit application code during this review; this skill is scoped to story
drafting artifacts.

When the review target is a tracked CONDAMAD story, updating
`_condamad/stories/story-status.md` is also allowed as a governance artifact.
Use the registry format defined by `condamad-story-writer`.

Recommended review artifact structure:

```md
# CONDAMAD Story Draft Review

## Review target
## Inputs reviewed
## Review layers
## Findings
## Acceptance audit
## Validation audit
## DRY / No Legacy audit
## Commands run by reviewer
## Residual risks
## Verdict
```

## Workflow Steps

Follow these steps in order:

1. `steps/step-01-target-and-context.md`
2. `steps/step-02-adversarial-review.md`
3. `steps/step-03-triage-and-evidence.md`
4. `steps/step-04-report-and-next-action.md`

At each step, include applicable regression guardrail IDs in the acceptance and
validation audit. A missing guard, missing baseline, or unreviewed allowed
difference for an applicable invariant is a finding.

## Non-Negotiable Rules

- Review mode is story-artifact first. Do not edit application code.
- Preserve the user's worktree. Do not revert unrelated changes.
- Use targeted story/capsule reads before drawing conclusions.
- Findings must include file/line evidence whenever possible.
- Findings must lead the final answer, ordered by severity.
- Do not report style-only noise unless it creates story ambiguity,
  validation risk, implementation-scope risk, or reviewability risk.
- Do not treat a skipped command as passing.
- Do not accept final evidence that lacks AC-to-validation mapping.
- Do not return `CLEAN` for an audit-sourced full-closure story while known
  in-domain residual work, broad allowlists, wildcard exceptions, unclassified
  fallback, compatibility, legacy, migration-only, shim, alias, TODO, or `PASS
  with limitation` remains.
- For No Legacy story contracts, ensure the story requires classification of
  legacy/search hits; missing classification requirements are review findings.
- If subagents are not explicitly authorized by the current user, run the
  adversarial layers yourself sequentially.
- If subagents are explicitly authorized, use them only for read-only review
  layers and keep final triage ownership in the main Codex session.

## Reviewer Command Policy

The editorial reviewer may run non-mutating commands, including:

- story validation and strict lint commands;
- targeted `rg` over story/capsule artifacts;
- targeted repository reads only when needed to verify story feasibility.

The reviewer must not run mutating application commands, including:

- formatters without `--check`;
- code generators;
- migrations;
- dependency updates;
- cleanup scripts;
- destructive git commands.

For Python repositories, prefer the repository command policy from `AGENTS.md`.
If a local venv exists, activate it when possible:

```powershell
# Windows PowerShell
.\.venv\Scripts\Activate.ps1
```

```bash
# POSIX shell
source .venv/bin/activate
```

When activation is uncertain, prefer invoking the venv interpreter directly:

```powershell
# Windows PowerShell
.\.venv\Scripts\python.exe -m pytest -q
```

```bash
# POSIX shell
.venv/bin/python -m pytest -q
```

In review mode, prefer check-only commands such as:

```powershell
ruff format --check .
ruff check .
pytest -q
```

Do not claim tests passed unless the exact command and environment were
recorded. Do not run `ruff format .` during review unless the user explicitly
asked to fix formatting.

## Final Verdict Values

Use exactly one:

- `BLOCKING`: Critical issue, High AC failure, security/data risk, active
  forbidden legacy, false validation claim, or missing validation evidence
  required by the story or definition of done.
- `CHANGES_REQUESTED`: material non-blocking defects require patching before
  acceptance.
- `ACCEPTABLE_WITH_LIMITATIONS`: no required patch found, but optional or
  full-regression validation could not be run and the residual risk is
  documented.
- `CLEAN`: no actionable findings and validation evidence is adequate.

For audit-sourced stories, `ACCEPTABLE_WITH_LIMITATIONS` is forbidden when the
story objective, ACs, source finding, or regression guardrails require full
closure. Use `CHANGES_REQUESTED` or `BLOCKING` until closure evidence is
complete, or document the blocker explicitly.

## Story Status Synchronization

After determining the final verdict for a tracked CONDAMAD story, synchronize
`_condamad/stories/story-status.md` before the final response:

- locate the row matching the reviewed story by `Story ID`, story key, or
  capsule `00-story.md` path;
- set `Status` to `done` only when the verdict is `CLEAN`, or when the verdict
  is `ACCEPTABLE_WITH_LIMITATIONS` and all required validation evidence is
  complete with only documented residual risk remaining;
- keep or set `Status` to `ready-to-review` when the verdict is `BLOCKING` or
  `CHANGES_REQUESTED`;
- never set `done` when required validation evidence is missing, an AC fails, a
  blocking regression guardrail is unresolved, or a required patch remains;
- preserve the existing `Story ID`, story key, title, path, and source;
- update `Last update` to the current local date in `YYYY-MM-DD` format;
- if the tracking document is missing but a capsule exists, create it using the
  `condamad-story-writer` registry format and seed the reviewed story row.

Do not report a story as accepted or done in chat or in
`generated/11-code-review.md` without also updating the matching registry row.

If the current review artifact already has a clean verdict and the requested
pass finds no new issue, do not rewrite the artifact only to record the pass.
Report the clean reuse in the final response.
