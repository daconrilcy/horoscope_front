## Ce qui a bien marché

1. Diagnostic en conditions réelles
- Les checks via `curl` ont mis en évidence les écarts runtime que les tests unitaires ne capturaient pas.

2. Isolation de validation
- L’usage de DB dédiées de validation a supprimé les faux positifs liés aux effets de bord des suites de tests.

3. Correctifs ciblés, delta minimal
- Le correctif front a été limité au contrat de données et au rendu défensif, sans refactor global.

4. Rigueur seed/runtime
- Le verrouillage de l’ordre de seed (`use_cases -> chapter 29 -> story 30-8 -> personas`) a rétabli la cohérence.

---

## Points d’attention

1. Couplage implicite entre scripts de seed
- Un seed générique pouvait écraser une configuration v3 sans garde explicite.

2. Écart contrat API vs contrat UI
- `disclaimers` déplacé côté API mais encore attendu dans certains chemins UI.

3. Validation “tests only” insuffisante
- Certaines anomalies n’apparaissaient qu’en exécution réelle complète.

---

## Actions retenues

1. Standardiser les garde-fous de seed
- Chaque seed qui met à jour `output_schema_id` doit vérifier la version actuelle avant overwrite.

2. Check de cohérence post-seed obligatoire
- Ajouter une vérification automatisée `use_case -> schema` pour les use-cases critiques.

3. Contract-tests front/back ciblés story
- Pour chaque story critique, ajouter un test qui valide la forme JSON réelle (champ top-level vs nested).

4. Smoke runtime systématique avant clôture
- Exiger un run `curl` de parcours nominal + erreurs métier sur endpoints critiques.

---

## Décision de clôture

Epic 30 est clôturé avec rétro réalisée.  
La story 30-8 est considérée stable en conditions réelles sur le périmètre validé (backend + frontend interpretation flow).
