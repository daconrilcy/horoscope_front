# Story 66.23: Normalisation canonique des identifiants de feature sur le domaine natal

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a Platform Architect,
I want to define and enforce a single canonical feature identifier for the natal domain across runtime, admin, seeds, coverage, and observability,
so that prompt assembly resolution, execution profile lookup, publication validation, seeds, tests, and telemetry all rely on the same taxonomy and no residual naming drift remains between `natal` and `natal_interpretation`.

## Contexte

Le chemin nominal natal doit utiliser `feature="natal"` et une variation métier portée par `subfeature`, puis suivre le pipeline standard `feature/subfeature/plan`.

Les stories 66.15, 66.20, 66.21 et 66.22 ont déjà convergé vers ce modèle, mais le code et les artefacts contiennent encore des usages historiques de `natal_interpretation`. Certains sont légitimes comme `use_case` contractuel ou nom de service/fichier, mais aucun ne doit rester une valeur nominale de `feature`.

Règle canonique cible :

- domaine fonctionnel : `feature="natal"`
- variation métier intra-domaine : `subfeature`
- variation contractuelle forte : `use_case` distinct seulement si le schéma de sortie change réellement
- compatibilité historique : alias/mapping explicitement déprécié, non nominal, télémétré, jamais source primaire

Phrase documentaire obligatoire à intégrer dans `docs/llm-prompt-generation-by-feature.md` :

> La feature canonique du domaine natal est `natal`. La désignation `natal_interpretation` ne doit plus être utilisée comme identifiant de feature nominal ; les variations métier relèvent du `subfeature` ou du `use_case` selon la nature de l'écart.

## Acceptance Criteria

1. **AC1 - Canonical feature identifier** : Le domaine natal utilise `feature="natal"` comme unique identifiant de feature canonique sur tout chemin nominal runtime, admin, publication, seed et observabilité.
2. **AC2 - Elimination of nominal `natal_interpretation` as feature** : Aucun appel nominal, aucune assembly publiée, aucune configuration nominale publiée ou publiable, aucune fixture nominale, aucun seed nominal, aucune métrique nominale et aucun exemple documentaire canonique ne doivent utiliser `feature="natal_interpretation"`.
3. **AC3 - Resolution consistency** : Toute résolution assembly ou waterfall concernant le domaine natal part de `feature="natal"` puis d'un `subfeature` canonique. Il ne doit plus exister de divergence de lookup entre un chemin résolu en `natal` et un autre résolu en `natal_interpretation`.
4. **AC4 - Compatibility governance** : Si un alias historique `natal_interpretation -> natal` est conservé temporairement, il est centralisé, classé non nominal, télémétré, interdit comme source de publication nominale, ne sert jamais de clé primaire de résolution nominale dans les registries, et est documenté avec trajectoire d'extinction.
5. **AC5 - Admin and publication lock** : Les validations d'admin et de publication refusent toute nouvelle configuration nominale utilisant `feature="natal_interpretation"` comme identifiant primaire de feature.
6. **AC6 - Seeds and fixtures normalization** : Tous les seeds, fixtures, factories, jeux d'essai et données de couverture relevant du domaine natal sont normalisés vers `feature="natal"`, sauf tests explicitement marqués comme compatibilité legacy.
7. **AC7 - Observability normalization** : Les métriques, logs structurés, labels et dashboards ne fragmentent plus la lecture nominale entre `natal` et `natal_interpretation`. Les événements nominaux remontent sous `feature="natal"` uniquement.
8. **AC8 - Test coverage** : La suite couvre la résolution nominale `feature="natal"`, le rejet des configurations nominales `feature="natal_interpretation"`, le comportement de compatibilité si alias conservé, et l'unicité des labels d'observabilité côté natal.
9. **AC9 - Documentation update** : `docs/llm-prompt-generation-by-feature.md` explicite la décision canonique, supprime les formulations ambiguës, documente le statut de tout alias legacy résiduel, et garantit que tous les exemples nominaux natal utilisent `feature="natal"`.
10. **AC10 - No silent drift** : Aucune couche ne réécrit ni ne tolère silencieusement `natal_interpretation` comme feature nominale sans traçabilité explicite.

## Dev Notes

### Points d'attention

- Ne pas renommer aveuglément tous les symboles `natal_interpretation` : plusieurs occurrences désignent un `use_case`, un service, un router, une table ou un champ métier historique. La story interdit `natal_interpretation` comme **feature nominale**, pas nécessairement comme nom de fichier ou contrat existant.
- La normalisation de `feature` ne vaut pas renommage automatique des `subfeature`. Les `subfeature` natal doivent être traités explicitement selon la taxonomie cible décidée par cette story, puis appliqués de façon uniforme.
- La compatibilité existante `DEPRECATED_USE_CASE_MAPPING` dans `backend/app/prompts/catalog.py` mappe déjà les use cases historiques nataux vers `feature="natal"`. Si un alias de feature est ajouté, il doit être séparé de ce mapping pour éviter de mélanger compatibilité de `use_case` et compatibilité de `feature`.
- Le fallback `use_case-first` est déjà gouverné par 66.21 ; ne pas le réactiver comme solution rapide pour contourner la taxonomie.
- Les validations 66.22 sur les providers montrent le pattern attendu : une source de vérité centrale consommée par admin, publication et runtime, avec métrique de gouvernance.

### Project Structure Notes

- Respecter la stack existante : backend FastAPI/Python, dépendances dans `backend/pyproject.toml`, pas de `requirements.txt`.
- Conserver le plus petit delta cohérent : privilégier un module de taxonomie partagé et des appels de validation courts plutôt que des règles recopiées dans chaque couche.
- Pas de migration DB obligatoire attendue sauf si une contrainte persistée existante rend impossible le verrou de publication. Si une migration est ajoutée, elle doit être idempotente et ne modifier que les champs strictement nécessaires.
