# Story 66.22: Verrouillage des providers supportés entre admin et runtime

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a Platform Architect,
I want to align the admin surface, publication workflow, and runtime execution rules with the providers that are actually supported end to end,
so that the platform no longer exposes nominal provider choices that fall back silently to OpenAI, and every published execution configuration reflects a real executable runtime path.

## Constat de départ

Le document d'architecture précise que :
- les profils internes stables (`reasoning_profile`, `verbosity_profile`, `output_mode`, `tool_mode`) sont bien la source de vérité côté plateforme ;
- un `ProviderParameterMapper` existe ;
- mais, à date, **OpenAI est le seul provider effectivement supporté et exécuté par le gateway** ;
- les providers non réellement supportés retombent encore sur le chemin de compatibilité OpenAI / `resolve_model()` ;
- `_call_provider()` n'accepte aujourd'hui que `openai` comme chemin d'exécution effectif.

Cela crée un écart d'architecture : l'admin et les entités de configuration peuvent laisser croire qu'un provider non-OpenAI est sélectionnable nominalement, alors que le runtime ne sait pas l'exécuter de bout en bout.

## Objectifs et Règles cibles

- **Objectif** : Faire de la liste des providers nominalement sélectionnables une projection stricte des providers réellement exécutables de bout en bout. 
- **R1. Admin** : L’admin ne doit proposer comme providers sélectionnables que les providers réellement supportés nominalement. (`openai` visible ; le reste masqué ou désactivé).
- **R2. Publication** : Une configuration d’exécution référencée par une assembly nominale ne peut pas être publiée si son provider n’est pas supporté (erreur de validation exploitable).
- **R3. Runtime nominal** : Sur chemin nominal, si un `ResolvedExecutionPlan` contient un provider non supporté, le gateway doit échouer explicitement. Fini les fallbacks silencieux sur `OpenAI`.
- **R4. Compatibilité non nominale** : Tolérance possible uniquement si `is_nominal=false` avec télémétrie explicite, interdit en prod nominale.
- **R5. Source de vérité** : Un unique registre des providers nominalement supportés est utilisé par l'admin, la validation, et le gateway.

## Acceptance Criteria

1. **AC1 — Registre canonique des providers supportés** : Un registre canonique déclare les providers supportés nominalement par la plateforme. À date, ce registre contient uniquement `openai`.
2. **AC2 — Admin aligné sur le registre** : L’interface d’administration n’affiche comme options publiables que les providers présents dans ce registre. Un provider absent du registre ne peut pas être sélectionné nominalement.
3. **AC3 — Blocage de publication** : Une publication nominale échoue si la configuration résolue d’exécution référence un provider absent du registre des providers nominalement supportés. Cette règle s’applique :
   1. à la publication d’un `ExecutionProfile` destiné à un usage nominal ;
   2. à la publication d’une `PromptAssemblyConfig` nominale qui référence un `ExecutionProfile` non supporté ;
   3. à toute validation de cohérence runtime conduisant à une résolution nominale vers un provider non supporté.
4. **AC4 — Erreur runtime explicite** : Le gateway rejette explicitement toute tentative d’exécution nominale avec un provider non supporté. Le résultat ne doit plus passer par un fallback silencieux vers OpenAI.
5. **AC5 — Compatibilité bornée hors nominal** : Les compatibilités hors nominal ne peuvent être tolérées que sur des chemins déjà classés par la gouvernance des fallbacks (`legacy`, `dev`, `test`) et explicitement tracés comme `is_nominal=false`. Aucun nouveau fallback provider implicite ne peut être introduit.
6. **AC6 — Observabilité dédiée** : Une métrique unifiée avec labels qualitatifs est émise, permettant de distinguer : `event_type=publish_rejected|runtime_rejected|non_nominal_tolerated`, avec les propriétés `provider`, `feature`, `is_nominal`, et `environment`.
7. **AC7 — Documentation mise à jour** : Le document `llm-prompt-generation-by-feature.md` est mis à jour pour refléter que le support nominal dépend du registre explicite, que ce registre est consommé comme unique source de vérité par l'admin, la validation et le runtime, et que `openai` seul est exécutable.
8. **AC8 — Non-régression OpenAI** : Les parcours nominaux existants `chat`, `guidance`, `natal`, `horoscope_daily` continuent de fonctionner sans changement.
9. **AC9 — Unicité de la source de vérité provider** : L’admin, la validation de publication et le gateway consomment le même registre canonique, sans duplication de liste locale ni allowlist divergente.

## Dev Notes

- **Architecture Patterns** : Distinguer "provider modélisable" (valeur possible dans l'abstraction), "provider mappable" (avec un mapper existant formellement) et "provider nominalement supporté" (exécutable bout en bout). C'est ce dernier niveau qui dicte la liste dans l'admin et la validité au moment du publish et de l'exécution.
- **Constraints** : Le runtime ne doit plus contenir de substitution implicite. Il faut échouer explicitement le plus tôt possible lors de la résolution du plan au lieu de retomber sur un provider valide par défaut sous le capot.

### Testing Requirements

- **Tests unitaires/intégration** : 
  - Un test de validation où un profile ou une assembly liée à `anthropic` échoue à la publication.
  - Un test de gateway où un `ResolvedExecutionPlan` avec `provider=anthropic` lève une erreur explicite précoce sur chemin nominal.
  - Un test de validation non-régression OpenAI sur les cas nominaux.
  - Un test spécifique sur un chemin non nominal toléré (ex. dev/test) vérifiant l'émission explicite de la métrique avec le label `event_type=non_nominal_tolerated`.
