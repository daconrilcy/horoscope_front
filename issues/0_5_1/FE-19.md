# FE-19 — Billing front observability correlation (request_id breadcrumbs)

## Contexte

Injecter des breadcrumbs `billing:*` avec `request_id` (header) dans un Debug Drawer dev-only pour faciliter la corrélation avec les logs backend.

## Objectif

- Mettre en œuvre billing front observability correlation (request_id breadcrumbs) pour aligner le front sur l'API backend Stripe.
- Offrir une UX claire, résiliente et testable.

## Portée / Hors-périmètre

- **Dans le scope** : Billing front observability correlation (request_id breadcrumbs).
- **Hors scope** : pricing logic côté backend, routes Stripe Server/webhooks (déjà gérées au backend).

## Design & Implémentation

- Dossier(s) impactés: `src/services/billing/*`, `src/components/billing/*`, `src/pages/billing/*`, `src/routes/*` (à adapter).
- Tech: React + TS, Zustand (si store), Zod pour validation, MSW pour tests.

### Tâches de dev

- [ ] Intercepteur HTTP pour extraire `request_id` des réponses billing.
- [ ] EventBus + Debug Drawer (dev-only) listant les derniers événements `billing:*`.
- [ ] Lier aux actions checkout/portal/terminal.

### Accessibilité & i18n

- Labels et descriptions accessibles (ARIA si modal).
- Texte FR prêt pour i18n.

### Observabilité

- Logger `request_id` (si header) dans un EventBus local (dev-only) pour corrélation.

## Tests

**Unit**

- [ ] Extraction header → event.
- [ ] Rendu du Debug Drawer en dev.

**E2E**

- [ ] Déclencher un checkout → breadcrumb visible dans le drawer (dev).

## Critères d’acceptation (AC)

- Les breadcrumbs apparaissent en dev et listent `request_id` corrélé.
- Aucune fuite de ce module en prod.

## Sécurité

- Idempotency-Key générée côté front pour toutes mutations.
- Ne jamais exposer de clés secrètes Stripe (utiliser les endpoints backend).

## Risques / Mitigations

- Erreurs de whitelist d’URL retour: message explicite + fallback configuré.
- Flags désactivés: champs masqués proprement.
