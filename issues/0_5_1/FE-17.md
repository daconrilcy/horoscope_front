# FE-17 — Stripe Terminal dev console (simulator flow)

## Contexte

Page dev-only pour piloter le simulateur Stripe Terminal : connection token, création PaymentIntent, process sur reader simulé, capture, cancel, refund.

## Objectif

- Mettre en œuvre stripe terminal dev console (simulator flow) pour aligner le front sur l'API backend Stripe.
- Offrir une UX claire, résiliente et testable.

## Portée / Hors-périmètre

- **Dans le scope** : Stripe Terminal dev console (simulator flow).
- **Hors scope** : pricing logic côté backend, routes Stripe Server/webhooks (déjà gérées au backend).

## Design & Implémentation

- Dossier(s) impactés: `src/services/billing/*`, `src/components/billing/*`, `src/pages/billing/*`, `src/routes/*` (à adapter).
- Tech: React + TS, Zustand (si store), Zod pour validation, MSW pour tests.

### Tâches de dev

- [ ] Créer page `/dev/terminal` protégée (flag `VITE_DEV_TERMINAL=true`).
- [ ] Boutons: Connect (connection_token), Create PI (amount/currency/capture), Process (reader_id), Capture, Cancel, Refund.
- [ ] Afficher ids/status (PI, charges).
- [ ] Idempotency-Key obligatoire sur toutes actions.

### Accessibilité & i18n

- Labels et descriptions accessibles (ARIA si modal).
- Texte FR prêt pour i18n.

### Observabilité

- Logger `request_id` (si header) dans un EventBus local (dev-only) pour corrélation.

## Tests

**Unit**

- [ ] Vérifier présence Idempotency-Key sur mutations.

**MSW**

- [ ] Simuler endpoints: connect, create, process, capture, refund avec cas d’erreurs.

**E2E**

- [ ] Scénario nominal: create → process → capture → refund (simulateur).

## Critères d’acceptation (AC)

- Le flow simulateur est pilotable depuis l’UI dev et reflète les statuts.
- Les actions refusent de s’exécuter sans Idempotency-Key.

## Sécurité

- Idempotency-Key générée côté front pour toutes mutations.
- Ne jamais exposer de clés secrètes Stripe (utiliser les endpoints backend).

## Risques / Mitigations

- Erreurs de whitelist d’URL retour: message explicite + fallback configuré.
- Flags désactivés: champs masqués proprement.
