# FE-16 — Portal session with whitelist UX & fallback

## Contexte

Intégrer le Customer Portal : création de session avec `return_url` optionnelle, gestion élégante des erreurs de whitelist, fallback sur URL par défaut.

## Objectif

- Mettre en œuvre portal session with whitelist ux & fallback pour aligner le front sur l'API backend Stripe.
- Offrir une UX claire, résiliente et testable.

## Portée / Hors-périmètre

- **Dans le scope** : Portal session with whitelist UX & fallback.
- **Hors scope** : pricing logic côté backend, routes Stripe Server/webhooks (déjà gérées au backend).

## Design & Implémentation

- Dossier(s) impactés: `src/services/billing/*`, `src/components/billing/*`, `src/pages/billing/*`, `src/routes/*` (à adapter).
- Tech: React + TS, Zustand (si store), Zod pour validation, MSW pour tests.

### Tâches de dev

- [ ] `BillingService.createPortalSession({ return_url? })`.
- [ ] Bouton “Gérer mon abonnement” (Header ou Account) → ouvre modal avec `return_url` optionnelle.
- [ ] Sur 400 whitelist error: afficher bases autorisées (si renvoyées par backend) + CTA pour utiliser l’URL par défaut.

### Accessibilité & i18n

- Labels et descriptions accessibles (ARIA si modal).
- Texte FR prêt pour i18n.

### Observabilité

- Logger `request_id` (si header) dans un EventBus local (dev-only) pour corrélation.

## Tests

**MSW**

- [ ] 200 → redirection `portal_url` ok.
- [ ] 400 invalid return_url → message explicite + retry/fallback.

**E2E**

- [ ] Aller/retour Portal → retour vers /account.

## Critères d’acceptation (AC)

- L’ouverture du Portal fonctionne en retour normal.
- Les erreurs de whitelist sont explicites et ont un fallback.

## Sécurité

- Idempotency-Key générée côté front pour toutes mutations.
- Ne jamais exposer de clés secrètes Stripe (utiliser les endpoints backend).

## Risques / Mitigations

- Erreurs de whitelist d’URL retour: message explicite + fallback configuré.
- Flags désactivés: champs masqués proprement.
