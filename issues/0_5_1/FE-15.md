# FE-15 — Billing success/cancel hardening with session correlation

## Contexte

Rendre robustes les pages /billing/success et /billing/cancel : lecture de `session_id`, revalidation du paywall/plan, messages de confirmation/annulation et CTA de reprise.

## Objectif

- Mettre en œuvre billing success/cancel hardening with session correlation pour aligner le front sur l'API backend Stripe.
- Offrir une UX claire, résiliente et testable.

## Portée / Hors-périmètre

- **Dans le scope** : Billing success/cancel hardening with session correlation.
- **Hors scope** : pricing logic côté backend, routes Stripe Server/webhooks (déjà gérées au backend).

## Design & Implémentation

- Dossier(s) impactés: `src/services/billing/*`, `src/components/billing/*`, `src/pages/billing/*`, `src/routes/*` (à adapter).
- Tech: React + TS, Zustand (si store), Zod pour validation, MSW pour tests.

### Tâches de dev

- [ ] Implémenter `BillingSuccessPage`: parse `session_id` (query), afficher message et bouton vers dashboard.
- [ ] Revalider l’état d’abonnement (refetch du user/entitlements).
- [ ] Implémenter `BillingCancelPage`: message clair + bouton retry vers Upgrade.
- [ ] Ajouter tracking d’événements (dev-only) pour corrélation request_id si dispo.

### Accessibilité & i18n

- Labels et descriptions accessibles (ARIA si modal).
- Texte FR prêt pour i18n.

### Observabilité

- Logger `request_id` (si header) dans un EventBus local (dev-only) pour corrélation.

## Tests

**Unit**

- [ ] Parsing `session_id` requis.
- [ ] Appel revalidation sur success.

**MSW**

- [ ] Simuler success et cancel et vérifier rendus.

**E2E**

- [ ] Flow complet: checkout → success → dashboard affiche le nouveau plan.

## Critères d’acceptation (AC)

- La page success lit `session_id`, revalide le plan et confirme l’upgrade.
- La page cancel explique clairement et propose un retry.

## Sécurité

- Idempotency-Key générée côté front pour toutes mutations.
- Ne jamais exposer de clés secrètes Stripe (utiliser les endpoints backend).

## Risques / Mitigations

- Erreurs de whitelist d’URL retour: message explicite + fallback configuré.
- Flags désactivés: champs masqués proprement.
